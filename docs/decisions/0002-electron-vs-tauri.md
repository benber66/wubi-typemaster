# ADR-0002：选择 Electron 而非 Tauri

> 状态：已接受
> 日期：2026-06-06

## 背景

在桌面应用框架选择上，主要考虑 Electron 和 Tauri 两个方案。

## 对比

| 维度          | Electron        | Tauri                  |
| ------------- | --------------- | ---------------------- |
| 包体积        | 大（50-100MB）  | 小（几 MB）            |
| 内存          | 高（100-200MB） | 低（30-50MB）          |
| 启动速度      | 较慢（1-2s）    | 快（<1s）              |
| 后端语言      | Node.js         | Rust                   |
| 学习曲线      | 平缓            | 较陡                   |
| 生态          | 非常成熟        | 较新但快速增长         |
| 跨平台构建    | 简单            | 中等（需 Rust 工具链） |
| 中文 IME 兼容 | 良好            | 良好                   |
| Web 技术      | 任何前端框架    | 任何前端框架           |

## 决策

选择 **Electron**。

理由：

1. **个人维护成本低**：纯 JavaScript/TypeScript 技术栈，无需学习 Rust
2. **生态丰富**：better-sqlite3、electron-builder、electron-updater 等生态成熟
3. **跨平台构建简单**：electron-builder 成熟
4. **中文 IME 兼容良好**：实际测试多种五笔输入法稳定
5. **开发效率高**：HMR 流畅，调试方便

## 接受的成本

- 包体积大（接受度 OK：自用为主）
- 内存占用高（个人电脑资源充足）

## 何时重新评估

- 若用户量增长、性能成为瓶颈
- 若团队规模扩大、需要 Rust 性能优势
- 若需要 macOS App Store 上架（Electron 上架 App Store 较复杂）
